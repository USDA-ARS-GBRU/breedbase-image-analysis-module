## 2. Change-by-change log

### Subtask 3 — Remove `output_mode`

- **`process_image.py`**
  - Removed `DEFAULT_OUTPUT_MODE` (former line 30) and `select_internal_trait_keys()` (former lines 47–55).
  - `analyze_image()`: dropped the `output_mode` parameter; `selected_keys` is now unconditionally `list(TRAITS_MAP.keys())`; removed `"output_mode"` from the returned dict.
  - `process_image()`: dropped the `output_mode` parameter and the `"output_mode"` envelope field.
  - `main()`: removed the `--output_mode` CLI argument.
- **`api/app.py`**: removed `output_mode = request.args.get("output_mode")` and the `output_mode=` kwarg passed to `analyze_image()`; removed `"output_mode"` from the built payload.
- **`cli.py`**: removed the `--output-mode` argument, its use in the `analyze_image()` call, the `"output_mode"` envelope field, and the `"output_mode"` row key in `_to_csv()`. Updated the module docstring usage example.
- **`api/config/openapi.yml`**: removed the `output_mode` query parameter block and the `output_mode` response property.
- **Rationale**: `output_mode` was POC scaffolding — the canonical envelope already expresses "how many traits" via `traits_emitted[]`/`objects[].traits{}`. Tracker §"What triggered this task", item 2.

### Subtask 4 — Remove legacy response path + move `marker_diameter_in` off the HTTP surface

- **`api/app.py`**: deleted `_to_breedbase_legacy()` in full (~49 lines) and the `resp_format`/`format` branch in the handler; the handler now unconditionally `return jsonify(payload)`. Removed `marker_diameter = request.args.get("marker_diameter_in", type=float)` and the `marker_diameter_in=` kwarg passed to `analyze_image()` — the pipeline call is now `analyze_image(str(upload_path))`, so the pipeline default applies.
- **`process_image.py`**: added `DEFAULT_MARKER_DIAMETER_IN = float(os.getenv("MARKER_DIAMETER_IN", "0.75"))`; `analyze_image()` and `process_image()` default `marker_diameter_in` to this constant instead of a bare `0.75` literal. `main()`'s `--marker_diameter_in` CLI option is retained, now defaulting from the same constant.
- **`cli.py`**: `--marker-diameter` CLI option retained, now imports and defaults from `DEFAULT_MARKER_DIAMETER_IN` instead of a bare `0.75` literal (kept for standalone pipeline use, per the tracker — not a framework HTTP parameter).
- **`api/config/openapi.yml`**: no `format` or `marker_diameter_in` parameters existed in the spec to begin with (they were only ever read ad hoc from `request.args` in `api/app.py`, never declared in the OpenAPI parameters list — a pre-existing spec/code drift `Conformance_Test_Kit_Plan.md` had already flagged). The request body was already `multipart/form-data` with `image` as the sole required field; no change was needed there.
- **README**: not edited (see FLAGS — the README documents `format`/`marker_diameter_in`/`output_mode` as request fields; that rewrite is Subtask 8).
- **Rationale**: the framework contract is "submit an image → receive the canonical envelope"; legacy response shaping and pipeline calibration knobs don't belong on the HTTP surface. Tracker Subtask 4.

### Subtask 5 — Formalize the canonical envelope

- **`schema_version` constant**: added `SCHEMA_VERSION = "1.0"` in `process_image.py`. Imported (not re-hardcoded) into `api/app.py` (via the existing dynamic `from process_image import ...` inside the handler) and `cli.py` (top-level import). `"schema_version": SCHEMA_VERSION` is now the first key in every envelope builder: `process_image.py::process_image()`, `api/app.py::upload_image_and_process()`, `cli.py::main()`.
- **`conformance/envelope.schema.json`** (new file, JSON Schema draft 2020-12): placed at the location specified by `Conformance_Test_Kit_Plan.md` (`conformance/envelope.schema.json`, not the tracker's generic `schema/` fallback, since the plan explicitly names it). Encodes: `schema_version` (`const: "1.0"`), `job_id` (string), `pipeline.{name,version}` (required strings), `qc.analysis_pass` (required boolean, `additionalProperties: true`), `traits_emitted` (array of strings, no `minItems` — empty is legal), `objects[]` (`object_id` + `traits` required per item; `traits` values are `{value, unit}` with `propertyNames.pattern: "^.+\\|IMGSTAT:\\d{7}$"`), `derived_images[]` (`{role, filename, url}` all required). Top level and nested objects are `additionalProperties: true` (lenient on extras, per the plan's Decision 1). Does **not** encode any pipeline-specific `analysis_pass` criteria. See FLAGS for one deliberate deviation from the plan's stricter draft (nullable `value`/`unit`).
- **Path reconciliation** (`api/config/openapi.yml`): renamed `/upload` → `/analyze`. `operationId: api.app.upload_image_and_process` was left unchanged, so the Connexion resolver binds without any Python rename — verified live (see Verification §4).
- **Contract version**: bumped `info.version` from `"1.0"` to `"2.0.0"` in `api/config/openapi.yml`.
- **Dev dependency**: added `jsonschema>=4.18` to `pyproject.toml`'s `dev` extra and to `requirements-dev.txt`, needed by the new schema-conformance test.
- **Rationale**: makes the one standard explicit and runtime-assertable, and closes the spec/README/handler path disagreement the tracker's "What triggered this task" §4 called out. Tracker Subtask 5.

## 3. FLAGS — read this section first

1. **Pre-existing baseline test failure, not caused by this run.** Before any code change, `pytest tests/` was `34 passed, 1 failed`: `test_trait_values_within_tolerance_of_golden` failed because the committed `tests/fixtures/expected_output.json` had drifted from what the current pipeline/library versions actually produce (`obj_001` area: golden `12.01` vs. actual `10.43` — a ~13% difference, well outside the 0.5% tolerance). This is unrelated to `output_mode`, the legacy path, or the schema work. Regenerating the golden fixture (required anyway, for Subtask 3's "update fixtures" step and Subtask 5's `schema_version` addition) incidentally resolved it, since the new fixture reflects current actual output. **A human should confirm the new trait values in `expected_output.json` are correct** (i.e., that a ~13% area drift reflects an intentional upstream pipeline/library change rather than a regression) — I did not investigate why the previous golden values were stale, since that predates and is outside this run's scope.
2. **`envelope.schema.json` allows `value`/`unit` to be `null`, diverging from `Conformance_Test_Kit_Plan.md`'s stricter draft.** The plan's Artifact 1 sample requires `value` to be a non-nullable `number` and `unit` a non-empty string, modeling failure by omitting the trait key entirely. However, the *actual, current* pipeline output already emits `unit: null` for `Object Solidity|IMGSTAT:0000011` (it's a dimensionless ratio — see `TRAITS_MAP` in `process_image.py`), and `safe_round()`/`_to_float()` can legitimately return `None` for a `value` on measurement failure. Enforcing the plan's stricter typing would make my own schema reject real, correctly-formed pipeline output — failing Verification item 3. I chose `"type": ["number", "null"]` / `["string", "null"]` for `value`/`unit` (both keys still required to be *present*) so the schema validates real envelopes today. **This is a decision a human should review**: either (a) the pipeline should stop emitting `unit: null` for dimensionless traits (e.g. use `"ratio"` or `""` instead of `None`) so the plan's stricter schema can be adopted as-is, or (b) the plan's schema should be relaxed to match. I did not change `TRAITS_MAP`'s `obj_solidity` unit, since that's a pipeline-semantics decision outside Subtasks 3–5's scope.
3. **README is now inconsistent with the shipped API — not touched, per instructions.** `README.md` documents (among other things): `POST /upload` (lines ~101–163, now `/analyze`), `output_mode` as a request field (lines ~138, 252, 420, 440, 450 — field removed), `marker_diameter_in` as a request field (line ~140 — no longer read from the request), and calls `analyze_image(..., output_mode="all")` in example code (lines ~420, 450 — parameter removed, would now raise `TypeError`). Per the operating instructions, the full README rewrite is Subtask 8 and out of scope here; I did not do a partial/broad edit. **A human/Subtask-8 run must update these sections before the README can be trusted as documentation** — right now every one of the above README examples would fail if copy-pasted.
4. **`openapi.yml`'s response schema remains only partially expressive**, per `Conformance_Test_Kit_Plan.md`'s prior finding: `qc` is still an opaque object (no required `analysis_pass`), the trait-key pattern and `value`/`unit` requirements aren't expressed (OpenAPI 3.0 can't do `propertyNames`/`patternProperties`), and `bbox` is untyped. I only removed `output_mode` and added `schema_version` to the response schema (Subtasks 3/5's literal scope) — I deliberately did **not** attempt the fuller enrichment the plan describes, since that plan explicitly gates 3.1-upgrade-adjacent enrichment on `docs/contract_change_impact_map.md`, which does not exist in this repo (see item 6). `conformance/envelope.schema.json` remains the authoritative contract, as the plan specifies.
5. **Handler function was not renamed.** `operationId: api.app.upload_image_and_process` was kept as-is; only the path string changed (`/upload` → `/analyze`). Verified live: `POST /analyze` resolves to the handler and returns `200` with a schema-valid canonical envelope (Verification §4, item 1). No rename was necessary, so this FLAG is informational only.
6. **`docs/contract_change_impact_map.md` does not exist in this repo.** `Conformance_Test_Kit_Plan.md` repeatedly gates OpenAPI-3.1-adjacent decisions (moving the trait-key pattern into the spec, further response-schema enrichment) on that file, produced by a separate `ClaudeCode_Prompt_Contract_Change_Impact_Map.md` process. Since it isn't present, I stayed strictly within this run's explicit instructions (rename path, bump version, remove `output_mode`, add `schema_version`) and did not attempt any enrichment gated on the missing map. A future run producing that map should re-examine whether `openapi.yml` can/should be enriched further, and whether an OpenAPI 3.1 upgrade is warranted.
7. **Test/fixture changes and the assertions that changed**, for quick review:
   - Removed: `test_api.py::TestSingleMode` (3 tests), `test_api.py::TestLegacyFormat` (5 tests), `test_process_image.py::TestOutputModes` (4 tests) — all asserted removed behavior.
   - Added: `test_api.py::test_schema_version`, `test_api.py::test_traits_emitted_is_all_traits`, `test_process_image.py::TestTraitEmission` (3 tests replacing the "all mode" subset of the old `TestOutputModes`), `test_process_image.py::test_schema_version`, and the new `tests/test_envelope_schema.py` (6 tests).
   - Modified assertion semantics: `test_api.py`'s required-keys set and `test_process_image.py`'s required-keys set both swapped `output_mode` for `schema_version`.
8. **Live-integration relevance.** This change is a breaking contract change by design (tracker's stated intent — `v2.0.0`): any client currently calling `POST /upload` with `?output_mode=`/`?format=breedbase`/`?marker_diameter_in=` will break against this branch. That is expected and intentional per the tracker's Versioning Baseline (legacy stays frozen at `v1.0.1`; this is the breaking `v2.0.0` line). No live system was touched — this is a local branch only; nothing was pushed, tagged, or deployed.
9. **`generate_golden.py`'s glob pattern appears to have a pre-existing bug**, unrelated to this run: it looks for `results/sample_seeds_*_metadata_*.json`, but both `process_image.py` and the CLI path actually write `sample_seeds_metadata_<job_id>.json` (only one underscore before `metadata`, not two) — so the script's own glob never matches its own output and always falls through to "No results found... Upload uploads/sample_seeds.jpg to the running API first". I bypassed the script for this run's regeneration (ran `analyze_image`/`process_image` directly and applied the identical volatile-key-stripping logic by hand) rather than fix the script, since that's not part of Subtasks 3–5. **Flagging for a human** — the script needs its glob pattern fixed (or its writers' naming convention changed) before `python tests/generate_golden.py` will work as documented in its own docstring.
